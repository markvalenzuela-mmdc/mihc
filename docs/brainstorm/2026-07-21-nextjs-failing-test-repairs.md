# Next.js Failing Test Repairs

## Context

The full Next.js test run reports failures in four areas after the application and
EnrollMate contract evolved:

1. The profile-navigation unit test renders an authenticated server page without
   mocking its authentication guard, which imports live database infrastructure.
2. The request-E2E-test action unit test does not mock the action's active-run
   database preflight.
3. Renderer tests expect an `external` option-source field and descriptions on
   fields where the current bachelors contract provides neither.
4. An integration test expects the historical fixed seed name `Ari Santos`, while
   seeded profiles are now generated by the EnrollMate mocker with approved UAT
   name markers.

The application behavior is correct. This repair updates test isolation and
expectations without changing production authentication, database, renderer,
contract, or seed-generation behavior.

## Agreed Approach

Use local test realignment. Each test file owns the mocks and fixtures needed for
the module it exercises. Do not add global auth or database mocks, and do not
restore obsolete production assumptions merely to satisfy tests.

Alternatives rejected:

- Global defaults in `__tests__/setup.ts` could hide missing dependencies in
  unrelated tests.
- Reintroducing an external contract field or a fixed seed name would change valid
  production behavior and make the contract less accurate.

## Unit Test Isolation

`profile-form-navigation.test.tsx` will mock `requireAuthenticated()` with a
representative authenticated user. The page can then serialize and pass both
EnrollMate flows without importing the real auth action or database client.

`request-e2e-test.action.test.ts` will mock `getDb()` with the minimum query shape
used by the action. The active-run lookup defaults to `null` before each test. A
new assertion will cover the existing-running-run response and confirm that no
Inngest event is published in that case.

These are unit-test boundaries only. Integration tests continue to use the real
test-database lifecycle.

## Renderer Contract Fixtures

The renderer supports free entry for both `external` and `cascade` option sources,
but the current bachelors definition does not contain an `external` field. Tests
will create a focused synthetic external field by cloning a real required field
that has a description and replacing its identity and option source. This exercises
the renderer branch without changing the captured contract.

Registered-control metadata assertions will follow the normalized contract:

- every listed required field must expose `aria-required="true"` and the required
  marker;
- fields with a description must expose that exact accessible description; and
- fields without a description must have no accessible description.

## Integration Seed Expectation

The seeded profile ID remains stable, while the faker-generated name is intentionally
variable. The route integration test will keep exact assertions for the response,
profile ID, and step count, then assert the documented generated-name pattern:
`SampleInternsTest`, `Interns`, or `Intern`, followed by a surname.

## Verification

Run the previously failing files first, then the complete Next.js test suite. Also
run TypeScript and ESLint checks. Success means:

- all Next.js tests pass;
- typechecking succeeds;
- lint reports no errors; and
- no production source file changes are included.

Existing lint warnings unrelated to these tests remain out of scope.
